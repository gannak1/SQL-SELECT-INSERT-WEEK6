# 저장 포맷 설계

## 왜 저장 포맷을 먼저 정해야 하는가
`INSERT`와 `SELECT`는 결국 파일에 데이터를 어떻게 저장하고 읽을지에 따라 구현이 달라진다.
따라서 SQL 파싱만큼이나 저장 포맷 설계가 중요하다.

## 1차 구현 전제
1차 구현에서는 여러 테이블을 동적으로 다루지 않는다.
하드코딩된 `users` 테이블 하나만 지원한다고 가정하고 저장 포맷을 설계한다.
즉, 지금 단계에서는 "범용 DB 파일 포맷"보다 "`users` 테이블을 안정적으로 저장/조회하는 방식"이 더 중요하다.

## 최종 선택: CSV 파일 + in-memory B-tree 인덱스
1차 구현의 실제 데이터 저장 포맷은 CSV로 유지한다.
하드코딩된 `users` 테이블 하나만 지원하므로 실제 데이터 파일은 `data/users.csv`를 사용한다.
그리고 `WHERE id = 값` 조회를 빠르게 하기 위해, 실행 중에는 `id -> CSV 파일 오프셋`을 담는 in-memory B-tree 인덱스를 함께 사용한다.

예:

```text
1,"kim01","Kim",25,"010-1234-5678","kim@example.com"
2,"lee02","Lee",21,"010-2222-3333","lee@example.com"
```

### 장점
- 사람이 직접 열어볼 수 있다.
- 디버깅이 쉽다.
- C에서 `fprintf`, `fgets`로 구현하기 편하다.
- `users` 테이블 하나를 빠르게 저장/조회하기에 충분하다.
- `WHERE id = 값` 조회는 B-tree 인덱스를 통해 전체 파일 스캔 없이 수행할 수 있다.

### 단점
- 타입 정보가 약하다.
- 컬럼 구조를 별도로 관리하지 않으면 해석이 애매해질 수 있다.

## 선택 이유
- 지금 단계의 목표는 범용 DB가 아니라 최소 SQL 처리기 구현이다.
- 하드코딩된 `users` 테이블을 다루는 데 CSV만으로 충분하다.
- 바이너리 포맷보다 구현, 디버깅, 발표 설명이 단순하다.

## CSV 컬럼 순서
`users.csv`의 한 행은 아래 순서로 저장한다.

1. `id`
2. `username`
3. `name`
4. `age`
5. `phone`
6. `email`

예:
- `1,"kim01","Kim",25,"010-1234-5678","kim@example.com"`

## CSV 저장 규칙
- 숫자 컬럼은 따옴표 없이 저장한다.
- 문자열 컬럼은 항상 큰따옴표 `"`로 감싸서 저장한다.
- 입력에서 문자열이 작은따옴표 또는 큰따옴표로 들어와도, CSV 저장 시에는 문자열 컬럼을 큰따옴표로 통일한다.
- 문자열 안의 공백과 쉼표는 값의 일부로 유지한다.
- 문자열 값 안에 큰따옴표 `"`가 있으면 CSV 저장 시 `""` 형태로 escape 한다.
- 문자열 내부 줄바꿈은 1차 구현에서 지원하지 않는다.

예:
- 입력 MiniSQL
  - `INSERT INTO users VALUES ('kim01', "Kim, Min", 25, '010-1234-5678', "kim@example.com");`
- 저장 CSV
  - `1,"kim01","Kim, Min",25,"010-1234-5678","kim@example.com"`

## 타입 해석 규칙
CSV 파일 자체는 텍스트 저장 포맷이므로 모든 값은 파일 수준에서는 문자 데이터로 저장된다.
하지만 프로그램은 `users` 테이블의 고정 컬럼 순서를 알고 있으므로 각 컬럼을 아래처럼 해석한다.

- `id`: 숫자 컬럼
- `age`: 숫자 컬럼
- `username`, `name`, `phone`, `email`: 문자열 컬럼

즉, CSV는 타입 정보를 스스로 저장하지 않고, 프로그램이 고정된 컬럼 정의를 기준으로 타입 의미를 부여한다.

## 헤더 행 사용 여부
1차 구현에서는 CSV 파일의 첫 줄에 컬럼명을 두지 않는다.
즉, 파일에는 실제 데이터 행만 저장한다.

이유:
- 구현이 더 단순하다.
- 컬럼 순서는 문서와 코드에서 이미 고정되어 있다.

## 파일 경로
- 데이터 파일: `data/users.csv`
- 인덱스 안내 파일: `data/users.idx`

## B-tree 인덱스 설계
- 인덱스 대상 컬럼은 `id` 하나이다.
- 인덱스의 실제 자료구조는 메모리 안의 B-tree이다.
- B-tree에는 `id`와 해당 행의 CSV 파일 오프셋(`ftell` 위치)을 저장한다.
- 프로그램 시작 시 `users.csv`를 한 번 스캔해서 B-tree를 재구성한다.
- `INSERT` 시에는 현재 최대 `id`보다 1 큰 값을 자동 생성한 뒤, CSV 파일 끝에 새 행을 추가하고 같은 `id`와 파일 오프셋을 B-tree에도 삽입한다.
- `SELECT * FROM users WHERE id = 값;` 는 B-tree로 먼저 위치를 찾은 뒤, 해당 한 줄만 파일에서 읽는다.
- `data/users.idx` 파일은 인덱스 사용 여부를 설명하는 안내용 파일이며, 실제 검색 자료구조는 메모리에 유지된다.

## C 입출력 방식
CSV는 텍스트 기반 저장 방식이다.
따라서 1차 구현에서는 아래 함수들을 중심으로 파일 입출력을 한다.

- `fopen`
- `fprintf`
- `fgets`
- CSV quoting을 고려한 문자열 분리 함수

## 비교 메모
아래 방식들은 현재 1차 구현에서는 채택하지 않는다.

### 스키마 파일 + 데이터 파일
예:
- `schema/users.schema`
- `data/users.csv` 또는 `data/users.dat`

### 장점
- 테이블 구조와 실제 데이터를 분리할 수 있다.
- 나중에 하드코딩을 제거하고 여러 테이블로 확장하기 쉽다.
- 컬럼 이름과 타입을 따로 설명할 수 있다.

### 단점
- 관리해야 할 파일이 2개가 된다.
- 구현 전에 스키마 포맷을 정의해야 한다.

### 데이터 파일 내부 헤더 + 데이터 영역
파일 앞부분에 메타데이터를 저장하고, 뒤에 실제 레코드를 저장하는 방식이다.

예시 개념:
- 파일 앞 4바이트: 컬럼 개수
- 다음 4바이트: 레코드 개수
- 그 뒤: 컬럼명/타입 정보
- 나머지: 실제 행 데이터

### 장점
- 파일 하나만으로 구조와 데이터를 함께 관리할 수 있다.
- 하드코딩을 줄이면서도 테이블 구조를 함께 저장할 수 있다.
- "파일 포맷을 직접 설계했다"는 차별화 요소가 된다.

### 단점
- 구현이 어렵다.
- 디버깅이 CSV보다 불편하다.
- 문서화가 더 자세해야 한다.

## 텍스트 저장과 바이너리 저장
모든 파일은 결국 디스크에 바이트 형태로 저장된다.
하지만 논리적으로는 아래 두 방식을 구분한다.

### 텍스트 저장
- 사람이 읽을 수 있는 형태
- 예: CSV
- 주로 `fprintf`, `fgets` 사용

### 바이너리 저장
- 프로그램이 정의한 구조대로 저장
- 사람이 바로 읽기 어려움
- 주로 `fwrite`, `fread` 사용

## 현재 확정 사항
- `users` 테이블 하나만 지원한다.
- 데이터 파일은 `data/users.csv`를 사용한다.
- 저장 형식은 CSV 텍스트이다.
- `WHERE id = 값` 조회에는 메모리 B-tree 인덱스를 사용한다.
- CSV 헤더 행은 사용하지 않는다.
- 컬럼 순서는 문서와 코드에서 고정한다.
- 숫자 컬럼은 따옴표 없이 저장한다.
- 문자열 컬럼은 항상 큰따옴표로 저장한다.
